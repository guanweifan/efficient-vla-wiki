## Claim
- 页面定位：这是一篇 **VLA RL training infrastructure / asynchronous systems** 论文；它的核心贡献是把 RL 训练流水线改写成全异步系统，而不是提出新的 policy architecture 或 inference-time acceleration 模块。
- 这篇论文要解决的是：现有基于强化学习的 VLA 后训练框架虽然能改善泛化，但训练流水线通常仍然是同步执行的，环境交互、rollout 生成和 actor 更新彼此串行依赖，导致 GPU 长时间空转、吞吐受限，特别是在 simulator 与模型训练争用资源时更明显。
- 核心主张是：如果把 VLA 强化学习训练流水线从 environment interaction、rollout generation 到 actor update 全部改写成分层异步系统，并进一步引入细粒度异步交互与 streaming micro-batch 更新，就能显著提升训练吞吐，同时基本保持训练成功率曲线和最终性能。
- 作者提出 `RL-VLA3`，强调三个层次的异步：`Train Async`、`Rollout Async` 和 `Streamer`。
- headline 数字需要拆开理解：
  - `59.25%` 对应 `LIBERO` 上相对同步/既有 decoupled 策略的吞吐提升；
  - `126.67%` 来自进一步优化 separation strategy 与 GPU allocation ratio 的更激进配置；
  - `8` 到 `256` GPUs 的结果则属于扩展性口径；
  - success-rate curves 与 baseline 基本保持一致，说明这首先是一篇“效率-性能保持”的系统论文，而不是任务性能突破论文。来源：[[raw/2602_05765_RL-VLA3.pdf]]，第 1 页摘要；第 6-8 页 Table 1、Fig. 2-3；附录 B.1。
- 更稳的主张是：`RL-VLA3` 试图把 VLA 强化学习后训练从同步执行改写成全异步基础设施，以提高训练吞吐和 GPU 利用率；其主要成果属于 training throughput / scaling 行为，而不是 inference latency 或 policy 质量新上界。

## 待核点
- 这篇的核心贡献是训练基础设施与系统吞吐，不是推理延迟或 policy architecture 本身；后续如果进入效率 taxonomy，需要避免把它误归为 inference acceleration。
- `59.25%` 与 `126.67%` 都是 throughput 口径，分别对应不同 baseline/placement ratio/async 组合，不能压成单一无条件提升。
- 论文提到 real robot deployments，但正文主结果仍然以 `LIBERO` 与 `ManiSkill` 的 RL training infrastructure 实验为主；“real-world validated”需要比 benchmark throughput claim 更保守。
- success-rate curves 被描述为“similar trend / minimal impact on final performance”，但不是全面优于 baseline 的任务性能论文；后续若写性能结论，需要突出这是效率-性能保持，而不是任务成功率突破。
- 若后续把它放进一般 `RL finetuning` 路线，需要继续保留“这是训练系统论文，不是策略方法论文”这一边界。
