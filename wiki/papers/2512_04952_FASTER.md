## Claim
- 页面定位：这是一篇 **autoregressive VLA neural action tokenization + policy design** 论文；它的核心贡献是围绕 action tokenizer 与 AR decoding 共同优化效率和性能，而不是单一 runtime trick。
- 这篇论文要解决的是：autoregressive VLA 虽然在高维连续动作建模上有统一生成优势，但其效率和性能高度受 action tokenizer 质量制约；现有离散 action tokenization 往往要么压缩率不足、推理慢，要么重建质量差、拖累策略性能。
- 核心主张是：如果把 action chunk 重新表示成 single-channel image，并用 neural tokenizer 学习非均匀的动作 patch 分组与离散压缩，再配合 block-wise autoregressive decoding 与 lightweight action expert，就能同时提升 AR-VLA 的动作重建质量、任务成功率和推理效率。
- 作者提出 `FASTER`，包括 `FASTerVQ` tokenizer 与 `FASTerVLA` policy。
- headline 数字需要拆开理解：
  - `112 ms`、`176 ms`、`197-556 ms` 是共享 `PaliGemma` backbone 下的 runtime 对比；
  - `95.5 -> 96.7 -> 97.7` 与 `323 ms -> 140 ms` 来自 `BAR` 与 `AE` 的 ablation；
  - `17.3%`、`96.65` 这类 tokenizer headline 又属于跨 backbone / tokenization 结果，不应与前述 latency 数字压成单一 superiority claim。来源：[[raw/2512_04952_FASTER.pdf]]，第 1 页摘要；第 7-8 页 efficiency tables；第 11-12 页 ablation。
- 更稳的主张是：`FASTER` 同时是一条 neural action tokenization 路线和一条 autoregressive policy design 路线；它的主要收益来自 tokenizer 质量、block-wise AR decoding 与 action expert 的组合，而不是单一组件的 isolated gain。

## 待核点
- 论文没有一个统一适用于所有 benchmark 的单一 headline；`112 ms`、`17.3%`、`95.5 -> 96.7 -> 97.7` 分别来自不同设置，后续不能压成一个无条件结论。
- `112 ms` 等 runtime 结论依赖共享 `PaliGemma` backbone、PyTorch 实现与 `RTX 5090` 硬件条件，不能直接写成一般性的端到端部署速度。
- 论文明确指出大部分性能提升来自 `FASTerVQ` tokenizer，本体 `BAR + AE` 进一步补充速度和性能；后续需要决定页面主轴是 tokenizer 创新还是完整 `FASTerVLA` 系统。
- `FASTer` 既是 tokenizer paper，也是 autoregressive policy paper；如果后续进入更高层 taxonomy，需要决定它主要归入 tokenization 路线还是 AR decoding / policy design 路线。
- 若后续把它并入一般 `AR-VLA efficiency` 路线，需要保留“核心是神经动作 tokenization + policy 联动设计，而不是单一速度优化”这一边界。
