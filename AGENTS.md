# AGENTS.md

## 1. 项目定位
1. 本文件定义本仓库根目录及全部子目录的人机协作与写作规则，用于约束 agent 如何组织知识、回答问题与维护结构。
2. 本项目的核心目标是：围绕 `raw/` 中的原始论文材料，构建一个**可追溯、可复核、可持续演化**的论文知识库；其中 `wiki/` 沉淀稳定知识，`threads/` 承载具体问题、推理过程、分歧与阶段性结论。
3. 本项目的最终价值不是“写得多”，而是“结论能回到原文、结构能持续维护、回答能经得起复核”。
4. 规则优先级始终为：**用户当前明确指令 > 本文件 > `WIKI_SCHEMA.md`**。

## 2. 基建底座
1. `raw/` 是原始论文层，只存放 PDF 等源材料，默认只读，并且是仓库内唯一事实来源。
2. `wiki/` 是持久知识层，由 agent 负责创建、维护、重构与交叉引用；只有经过原文核证的稳定结论才能进入 `wiki/`。
3. `threads/` 是问题讨论层，用于记录问题、回答、推理、中间判断、比较、冲突、未决项与阶段性结论；它可以承载分析，但它本身不是事实来源。
4. 解析器输出、文件名、历史讨论、旧页面、旧 taxonomy 或任何外部网页/仓库/常识，只能作为定位线索或工作中间层，不能替代 `raw/` 证据。
5. `WIKI_SCHEMA.md` 属于结构规则层，用来描述 wiki 的组织方式、字段与维护流程；它不是事实来源，但 agent 应持续主动维护它，使之与当前 wiki 结构保持一致。

## 3. 风格规范
1. 所有事实性表述都必须以 `raw/` 为依据，不得把推测、印象、常识或未经核证的历史内容写成事实。
2. 回答与写作应优先追求证据清晰、边界明确、表达克制；不要为了“完整”而补写作者未明确给出的观点、实验设置、因果解释或横向结论。
3. 当证据不足时，必须显式标注“未提及”“不确定”或“待核”，不能模糊带过，更不能凭空补全。
4. 当不同论文之间存在冲突时，应先记录冲突与出处，不强行统一口径，也不擅自裁决谁更正确。
5. 结构与表达都应追求简约：优先保留最小但足够的信息单元，避免空泛总结、重复摘录、过度分类与形式化堆砌。

## 4. 当前重心
1. 默认工作流是：**先读 `wiki/` 建立上下文，再在结论关键、细节敏感、存在冲突或 `wiki` 不足时回到 `raw/` 复核。**
2. 回答问题时，应优先利用既有 `wiki/`，但任何关键判断都要以是否能回到原文为准，而不是以 wiki 现状为准。
3. 若 `wiki/` 结构已经妨碍回答质量、检索效率或后续维护，应先调整 `wiki/` 与 `WIKI_SCHEMA.md`，再继续回答 `threads/` 或补写内容。
4. 当前阶段的核心任务是持续把问题讨论转化为结构化知识，把已核证的稳定结论沉淀进 `wiki/`，把未定结论留在 `threads/`，并保持两者边界清晰。
5. 若 `raw/` 中证据不足以支持目标结论，应暂停扩写，先说明证据缺口，而不是勉强完成表面任务。

## 5. 质疑驱动
1. 若对任务目标、上下文、证据状态或用户前提存在疑问，必须及时指出，严禁在理解模糊的情况下盲目执行。
2. 若发现用户问题中的前提与 `raw/`、既有 `wiki/` 或当前结构存在冲突，应先说明冲突，再决定是否继续推进。
3. 质疑的目的不是拖延，而是确保交付物真正可用、可核、可维护。

## 6. 追求简约
1. 始终寻找更简单、更本质的组织方式与回答路径，优先选择能降低维护成本、减少重复劳动的方案。
2. 若已有结构足以承载信息，就不要新增层级、页面、字段或流程；若存在更优的简化方案，应主动提出并说明理由。

## 7. 目标闭环
1. 开始执行前，应先把任务转化为可验收目标，例如需要补哪些页面、核哪些证据、回答哪些问题、保留哪些未决项。
2. 完成任务时，应对照目标逐项检查，明确哪些已经落实、哪些仍待核、哪些因证据不足而暂停。
3. 任何新增内容都应说明其应落在 `wiki/`、`threads/` 还是仅作为中间过程存在，避免知识状态混乱。

## 8. 拒绝冗余
1. 严禁过度设计，不创建没有实际维护价值的分类、模板、索引或总结段落。
2. 不重复搬运同一份证据；确需复用时，优先通过交叉引用、摘要重述或结构复用来减少重复。
3. 不为尚未出现的问题预留大段兜底结构；规则、页面与字段都应以真实需求驱动。

## 9. 独立思考
1. 应从原文证据和任务目标出发独立判断，不被历史写法、旧 taxonomy、解析器产物或未经核证的用户先验牵着走。
2. 即使用户给出了方向，也应主动判断其中哪些是需求，哪些只是未经验证的假设；在不违背用户明确指令的前提下，坚持基于证据修正路径。

## 10. 规则迭代
1. 若识别到用户反复提出的偏好、约束或验收标准，应主动提议将其沉淀为规则，推动本文件持续演化。
2. 若 wiki 的结构、字段或维护流程出现系统性变化，agent 应同步更新 `WIKI_SCHEMA.md`，不必等待用户介入。
3. 若只是局部临时策略，不要轻易写入长期规则；规则更新应服务于长期稳定性，而不是一次性方便。

## 11. 复盘机制
1. 对重复出现的错误，应归纳其根源，明确是证据核对不足、结构设计失当、表达越界，还是流程执行失真。
2. 复盘不应停留在口头层面；必要时应将可复用的修正经验沉淀为 `AGENTS.md` 或 `WIKI_SCHEMA.md` 的增量规则。

## 12. 授权逻辑
1. `AGENTS.md` 由用户主导；agent 如认为本文件需要修改，必须先征得用户同意。
2. `WIKI_SCHEMA.md` 不受前条约束，agent 可在不违背本文件的前提下持续主动维护。
3. 涉及仓库核心结构、重要组织逻辑或明显改变用户协作方式的改动，必须先获得用户明确许可。
4. 局部修复、结构整理、证据补录与常规维护可由 agent 自主完成，但完成后应明确告知。

## 13. 分治协作
1. 若任务天然可拆解，应优先分解为若干边界清晰的子任务，以降低认知负担并提高推进效率。
2. 在当前工作流支持的情况下，可将互不冲突的子任务并行推进；主 agent 负责全局调度、结果整合、最终验收以及子 agent 的生命周期管理 。
3. 分治的前提是边界清晰与结果可整合，而不是为了形式上的并行而制造额外复杂度。
