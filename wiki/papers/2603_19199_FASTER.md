## Claim

- 页面定位：这是一篇 **flow-based VLA reaction-time optimization** 论文；其核心贡献是 reaction-centric sampling schedule 与 streaming execution 机制，而不是一般 VLA 的通用推理加速框架。
- FASTER 的核心命题是：现有 real-time flow VLAs 虽然通过 asynchronous inference 改善了轨迹连续性，但仍然忽视了 **reaction latency** 这一更关键的实时性维度；真正限制机器人对动态环境做出快速响应的，不只是整体 inference latency，而是 **Time to First Action (TTFA)** 以及 inference-execution cycle 的频率。
- 论文认为，flow-based action chunking 里广泛使用的 constant timestep schedule 会强迫所有 action 都完成同样多的 denoising steps，导致最紧急的 immediate action 也要等整段 action chunk 一起“去噪完”才能发出，这正是反应迟缓的根源。
- 为此，论文提出 FASTER：通过 Horizon-Aware Schedule (HAS) 优先加速 near-term actions 的 flow sampling，把 immediate reaction 压缩到 single-step sampling；再配合 streaming client-server interface 与 early stopping，让机器人在后续动作尚未完全生成时就先开始执行。
- headline claim 需要拆开理解：
  - `10x` 指的是 immediate reaction 所需 denoising 压缩幅度，而不是统一的端到端系统提速。
  - `3x TTFA boost` 与 reaction capability 改善主要来自特定 flow-based VLA、特定 GPU 与特定任务环境。
  - LIBERO / CALVIN 的结果更接近“在轻微性能代价下换取更快反应”，而不是无条件全面领先。
- 更稳的主张是：**要让 flow-based VLA 具备真实世界反应能力，必须优先优化 immediate action 的生成时刻，而不是只优化整段 action chunk 的平滑异步执行。**

## Methodology Index

- **Reaction-centric real-time framing**：把实时性重心从 smooth execution 转向 prompt reaction。
- **Time to First Action (TTFA)**：提出 TTFA 作为比整体 inference latency 更贴近 reactivity 的核心指标。
- **Horizon-Aware Schedule (HAS)**：按 action horizon 分配不同 denoising schedule，优先完成 near-term action。
- **Single-step immediate action generation**：让第一个动作在一次 AE sampling 后即可发出，而不是等整块 action chunk 全部完成。
- **Streaming client-server interface**：服务器侧逐步发送已完成动作，客户端边收边执行，不再等待完整 chunk。
- **Early stopping for execution horizon**：只要 execution horizon 内的动作已完成，就跳过剩余不必要 sampling steps。

## 待核点

- `10x`、`3x`、TTFA 改进、以及实机任务优势目前仍是 bundled headline，后续需要分别拆到模型、GPU 平台与任务设置。
- 论文把 HAS、streaming client-server、early stopping 一起包装成 FASTER 主体；后续 evidence 层可能需要区分“sampling schedule 本身”的贡献与“系统工程式 streaming / stopping”的贡献。
- 论文强调方法对 flow-based VLAs 是 plug-and-play，但当前直接证据主要集中在 `π0.5` 与 `X-VLA`；后续需要明确它对其他 flow-based VLA 乃至非-flow action chunking policy 的外推边界。
